# 隐私与数据边界说明

## 原则

AI Agents Leader 的目标是做本地 AI Runtime Signal System，不是远程托管平台。

默认设计原则如下：

- 优先本地运行
- 优先本地状态判断
- 优先最小化采集
- 不把用户工作内容当作产品数据资产

项目的核心目的是判断 agent 是否在运行、等待确认、已完成、出错或可能卡住，而不是收集完整对话内容。

## 当前会接触到哪些本地信息

为了完成状态检测，项目当前可能会接触以下本地信息：

- 本地进程信息
  - 用于判断特定 agent 或相关宿主进程是否存在
- 本地会话或状态文件
  - 用于解析 agent 是否处于 thinking、running、completed 等状态
- 本地工作目录信息
  - 某些 adapter 会把目录作为状态元信息展示在 UI 中
- 本地端口占用情况
  - 用于 runtime、HTTP API、WebSocket、Overlay 的启动与发现
- Claude Code hooks 相关本地插件目录
  - 用于安装和触发状态上报 hooks

## 默认不会做什么

在当前设计下，项目默认不会主动做以下事情：

- 不上传完整对话内容到项目维护方服务器
- 不上传代码仓库内容到项目维护方服务器
- 不采集与你的 agent 状态无关的系统数据
- 不把本地文件做云端持久化
- 不内置远程遥测或分析服务

如果未来引入任何可选遥测、崩溃报告或远程诊断能力，必须满足两点：

- 默认关闭或明确告知
- 在文档中写清采集范围、目的与关闭方式

## 网络边界

当前项目内置的网络通信主要是本机回环地址上的本地通信：

- WebSocket：用于 runtime 与 overlay 同步状态
- HTTP API：用于状态上报、健康检查、agent 查询

默认端口：

- WebSocket：9988
- HTTP API：9989
- Overlay UI：1666

这些能力的设计目标是本地进程之间通信，而不是对外暴露公共服务。

在公开发布时，建议使用者注意：

- 不要把本地调试端口直接暴露到公网
- 不要在不可信网络环境下做额外端口转发
- 如果自行扩展第三方 adapter，请自行评估其网络与权限边界

## UI 中可能显示的信息

当前 UI 可能展示以下内容：

- agent 名称
- 当前状态
- 最后活动时间
- 运行耗时
- 工作目录

这意味着在录屏、截图或直播时，工作目录等元信息可能被看到。

如果后续需要更强的隐私保护，适合增加的能力包括：

- 隐藏目录
- 目录脱敏
- 演示模式
- 红录屏模式

## Hooks 与 Adapter 的注意事项

某些 adapter 会依赖本地 hooks、文件监听或进程检测。它们的风险边界通常在于：

- 是否读取了超出状态判断所需的内容
- 是否把过多元信息暴露到 UI
- 是否在错误场景下留下本地残留文件或进程

因此，新增 adapter 或扩展现有 adapter 时，建议遵守以下要求：

- 只读取完成状态判断所必需的数据
- 优先提取状态信号，而不是原始内容全文
- 文档必须写明读取来源、用途和副作用
- 涉及安装 hooks 或创建文件时，要写明安装路径和卸载方式

## 安全与隐私不是一回事

安全问题请参考 [SECURITY.md](SECURITY.md)。

本文件关注的是：

- 项目会接触哪些本地数据
- 默认不会如何处理这些数据
- 使用者在公开环境或扩展场景下需要注意什么

## 后续开源建议

在项目正式大规模公开前，建议继续补充：

- 哪些 adapter 会读取哪些文件
- 每类状态来源的最小数据需求
- hooks 的安装、卸载、禁用说明
- 演示模式或隐私遮罩策略

这样外部贡献者和使用者才能更容易判断项目的实际边界，而不是靠猜。
